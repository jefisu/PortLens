PortLens · Specs v1.1

Portfolio clarity, tax-aware.

Single-source specification for PortLens — a desktop tool that helps Brazilian retail investors run the monthly tax-exemption loophole (up to R$ 20,000 of equity sales / month, per Lei 11.033/2004) to raise their average cost basis and reduce future capital-gains tax. This document bundles the product brief, the visual & interaction guidelines, and the eleven canonical screens.

How to read this file. The sidebar groups the spec into three blocks: Product (what the tool does), Design Guidelines (how it looks and behaves), and Screens (the canonical mockups). The product UI inside the screens is in Portuguese because that is the shipped language; the spec narrative around it is in English.
Product

Context

PortLens is a portfolio-tracking tool for individual Brazilian investors using the monthly tax-exemption strategy — commonly called "limpeza de lucro" (gain cleanup) — to legally raise their stock cost basis. Brazilian tax law (Lei 11.033/2004) exempts from income tax any capital gains on monthly equity sales totaling up to R$ 20,000 (across all tickers combined). The product's single job is to tell the user, at any moment: "Can I sell this position right now without paying tax?"

The user records every transaction (manual entry, CSV, or PDF brokerage note). PortLens computes weighted average cost, accumulated monthly sale volume, realized gains/losses, and the current exemption status in real time. The design is dark-first, data-dense, and keyboard-driven because the target user is comfortable with spreadsheets and comes to PortLens to look at numbers, not to be delighted by animations.

Module 1 — Transactions (Lançamentos)

Record of every portfolio operation. Each entry contains: date, ticker, type (Compra / Venda / Recompra), quantity, and unit price. From these entries the system derives, per operation and in real time:

  • Operation volume
  • Weighted average price before the operation (historical weighted cost basis)
  • Quantity and total cost held before and after
  • New average price after the operation
  • Realized gain or loss (only on sales)
  • Accumulated sale volume in the current month (summed across every ticker)
  • Exemption status for the month (≤ R$ 20,000 = exempt)

Module 2 — Current Portfolio (Carteira Atual)

Consolidated view per ticker showing:

  • Current position and weighted average price
  • Market value (using the user-provided quote)
  • Unrealized gain and return
  • Suggested sale quantity to stay within the monthly exemption limit

Module 3 — Monthly Summary (Competências)

One row per month/year with:

  • Total sale volume for the month (all assets combined)
  • Exemption status
  • Realized gain for the month
  • Estimated income tax (15% of the gain, only when taxable)
  • Total quantity sold and repurchased

Terminology note: Competência is the Brazilian fiscal period (usually a calendar month). We keep the term in Portuguese because it is the label the product surfaces to users and it has no clean one-word English equivalent.

Critical business rules

  1. The R$ 20,000 limit is monthly and sums across every ticker sold in that month, not per asset.
  2. The exemption applies only to regular stocks — not FIIs (real-estate funds), ETFs, or BDRs.
  3. A recompra (repurchase) must happen on D+1 to qualify as swing trade, not day trade.
  4. The new weighted average after a repurchase is the repurchase price weighted with the remaining balance.
  5. Even when exempt, the operation must still be declared in the IRPF (Brazilian income-tax return).

Data import

Beyond manual entry, PortLens accepts two kinds of input:

  • Brokerage note (PDF) — the official document issued by the broker. Operations are extracted automatically.
  • CSV file — data previously exported from PortLens itself, enabling backup and migration between environments.

The import flow is a three-step modal (file → review → confirm). Deduplication is anchored on the brokerage note number: if the number is already in the database, the flow halts at step 1 with a "note already imported" state. Nothing is persisted until the user explicitly confirms step 3.

Design Guidelines

1 · Principles

Five principles govern every interface decision in PortLens. When they conflict, the order defines the priority.

1. Fiscal clarity above everything

The user opens PortLens to answer a single question: "Can I sell this without paying tax?". The answer must be readable in under two seconds from anywhere in the app. The current-month exemption status is the most important datum in the product and deserves permanent real estate on screen.

2. Density with breathing room

Investors want to see many numbers at once — average cost, position, unrealized gain, accumulated volume, distance to the monthly limit. The design does not run from density; it organizes it. Generous cell padding, clear typographic hierarchy, and strict separation between data groups make a twelve-column table legible.

3. Data entry is the critical path

Logging a transaction should be a keyboard action, not a click journey. The entry form must be one keystroke away, accept sequential keyboard navigation, and deliver immediate feedback on derived values (new average, accumulated volume, exemption status) while the user types. Bulk import (PDF or CSV) is an extension of this principle: the system does the heavy lifting but the user always reviews and confirms before anything is persisted.

4. Numbers don't lie but need context

No monetary value appears without unit, sign (gain/loss), and — when relevant — distance to a limit. R$ 18,450 says nothing; R$ 18,450 of R$ 20,000 (92%) tells a story.

5. Dark by default, light with respect

Dark-first is not aesthetics, it is ergonomics for people staring at spreadsheets at night. But the light theme is first-class — users who need to print or work in direct sunlight deserve the same level of polish.

3 · Typography

Three families, each with a specific role. All are freely available on Google Fonts.

Display — Fraunces

Contemporary variable-axis serif. Used for large headings, the wordmark and hero numbers. Gives PortLens a voice between financial editorial and modern fintech — sophisticated without banking austerity or serif cliché.

Interface — Inter Tight

Sans-serif designed for UI, with generous x-height and excellent small-size readability. Lives harmoniously next to Fraunces without competing.

Numeric / Tabular — IBM Plex Mono

Monospace with active tabular variant, humanist drawing with slight warmth. Every monetary value, quantity and percentage uses this family — columns of numbers auto-align and visual comparison becomes instant.

Inviolable rules for numbers

  • Every monetary value is prefixed with R$, thousands separator dot, decimal comma — Brazilian standard.
  • Signed values (gains/losses) always show + or − (typographic minus, not hyphen).
  • Percentages use decimal comma and a suffix % without space.
  • Share quantities are integers, without thousands separator when below ten thousand.
  • Alignment: monetary values are always right-aligned in tables.
  • Tabular digits mandatory in any numeric container.

5.3 New-Transaction Form (side panel)

Opens as a 420px panel sliding in from the right. Never a centered modal — the side panel preserves the table context behind it.

Field order: Type (segmented: Compra / Venda / Recompra) · Date (default today) · Ticker (autocomplete on existing holdings) · Quantity · Unit price (R$ prefix).

Live-preview panel below the fields, background --bg-surface-2. Recalculates on every keystroke (150ms debounce) and shows: operation volume, PM before → PM after, monthly volume after operation with progress bar, and resulting fiscal status. This is the second signature element of the product — the user sees the fiscal impact before confirming.

Mandatory shortcuts: N (global) opens the panel · Tab moves forward · Enter confirms when valid · Esc closes without saving.

6.2 Micro-interactions

  • Saving a transaction: the new row slides into the table with a brief slide+fade animation (400ms) and the status card's progress bar transitions in 300ms.
  • Exemption status change (e.g. crossing the limit): the card pulses once (scale 1.0 → 1.02 → 1.0, 400ms).
  • Preview calculation: updated values flash for 200ms in --neutral-soft.

6.3 Focus states

Accessibility is not optional. Every interactive element gets a visible focus ring: outline: 2px solid --accent; outline-offset: 2px; radius inherits.

6.4 Empty states

  • Transactions table with no entries: thin-stroke illustration, text "Nenhum lançamento ainda", primary CTA "Registrar primeiro lançamento".
  • Month with no sales in Module 3: row with --bg-surface-2 background and — text in --text-disabled, no exemption badge.

6.5 Error states

Validation errors appear below the field, in --loss, size body-sm. Specific messages, not generic: "Data não pode ser futura" instead of "Valor inválido".

7 · Accessibility

  • Minimum AA contrast on all text. Secondary text in --text-secondary on --bg-base reaches 7.1:1 (AAA).
  • All interactive components navigable by keyboard.
  • Global shortcuts documented in a modal opened with ?.
  • Screen-reader announcements on the exemption card so changes are vocalized.
  • Explicit semantic table structure.
  • Focus never lost: closing the side panel returns focus to the button that opened it.
  • Color-blind safety: the exemption status never relies on color alone — it also uses icons (✓, ⚠, ✕).

8 · Voice & tone

PortLens's voice is direct, technical, and does not talk down. The user is an investor who knows what they're doing — the product doesn't explain what income tax is, it assumes you know.

  • Short sentences, active voice.
  • Numbers always explicit: "R$ 7.660,00 restantes" rather than "ainda resta margem".
  • No slang ("bora!", "show!") or decorative emojis.
  • Correct technical terms: "alienação" is acceptable in formal contexts, "venda" in everyday use. "Competência" instead of "mês" when referring to the fiscal period.
  • Legal notices are neutral: "Mesmo isento, declare no IRPF" — no exclamation, no excessive alert tone.

10 · Screen inventory & import flow

v1 has four main screens, accessed from the sidebar:

  1. Dashboard / Overview — entry page, gathers the exemption-status card, monthly metrics, and latest transactions.
  2. Lançamentos (Module 1) — full table of every operation, with filters by competência and ticker.
  3. Carteira Atual (Module 2) — consolidated per-ticker view with evolution chart and sale suggestions.
  4. Competências (Module 3) — month-by-month history with fiscal status, realized gain, and estimated tax.

Supporting states: the new-transaction side panel (overlaid on any screen) and the empty state (first run).

10.1 Import flow

Import is a three-step sequential flow rendered as a full-screen stepper modal (not a side panel — the data volume needs the whole viewport):

Step 1 — File selection. The user picks between "Nota de corretagem (PDF)" and "Dados exportados (CSV)". A drop zone is the only affordance — no text field. When a PDF is processed, the system extracts the brokerage-note number before anything else. If that number is already in the database, the flow halts before Step 2 and the "note already imported" state is shown (see §10.3). This prevents duplicate imports even when the user re-uploads the same file by mistake.

Step 2 — Review extracted operations. This is the central, most critical step. The system shows a table of every operation read from the file, headed by the note identification (number, issue date, broker). Each row has an extraction status column:

  • Pronto (● green): operation extracted confidently, no ambiguity.
  • Revisar (● yellow): low-confidence read or incomplete field — highlighted editable cell.
  • Conflito (● red): operation already exists via manual entry with the same fields — the user chooses ignore or replace. This case is residual when the note number is the primary deduplication key.

The user can edit individual cells before confirming. Footer reads: "X operações prontas · Y para revisar · Z conflitos".

Step 3 — Confirm and result. On confirm, the system processes operations in bulk, persists the note number tied to each operation, and recalculates PM, volume, and exemption status. The screen shows the result: how many were imported, which were skipped, and whether the exemption status of the affected month changed. A CTA takes the user back to Lançamentos with the new operations visible.

10.2 Traceability by Note Number

The brokerage-note number is a traceability metadata persisted next to each operation imported via PDF. Display and usage rules:

  • On the Lançamentos table, operations linked to a note show the note number in an optional column (hidden by default, toggled via column settings). When visible, it renders in --text-tertiary so it doesn't compete with primary data.
  • Hovering a row with a linked note shows a compact tooltip: note number, issue date, and broker.
  • Manually-entered operations have no note number — the cell shows — in --text-disabled.
  • The field is read-only — the user cannot edit it after import.
  • CSV export includes the note-number field, preserving traceability on future re-imports.

Note-already-imported state (halts flow at Step 1): the system shows a warning card with the note number, the date it was imported and how many operations it generated. Two options: "Ver os lançamentos dessa nota" (opens the filtered table) and "Cancelar" (closes the flow). There is no re-import option — the note is already in the base and its operations can be edited individually if needed.

10.3 CSV export

Export is not a screen — it's an action available on the Lançamentos page, as a secondary button in the header. It generates the file immediately, with no intermediate confirmation, and shows a success toast for 3 seconds. The exported file includes the note-number field and is the same format accepted by the importer, guaranteeing round-trip compatibility.

10.4 Import UX rules

  • Deduplication by note number. For PDFs, the note number is the primary dedup key — checked before any processing. Row-level conflicts are residual and only occur between manual entries and imports.
  • Never silently overwrite. Any residual conflict requires explicit user decision — the app never auto-resolves.
  • Preview before persist. No imported operation is saved before Step 3 confirmation. The user can abandon the flow at any time with zero side effects.
  • Read errors are explained. If the PDF cannot be read, the message names the specific problem: "Número de nota não encontrado no documento" or "Formato de nota não reconhecido" — not "Erro ao processar arquivo".
  • Partial import is valid. The user can uncheck individual rows before confirming, importing only a subset.
  • Immediate fiscal feedback. In the Step 2 footer, show the projected impact on the most-affected month's exemption status: "Após importar, Nov/2025 passará de R$ 12.340 para R$ 19.260 vendidos (ainda isento)."

11 · Quality checklist

Before calling a screen "done", validate:

  • Current-month exemption status is visible or one click away.
  • Every monetary value is in the monospace family with tabular digits.
  • +/− signs on gains and losses, with consistent semantic colors.
  • No value without unit or context.
  • Keyboard focus visible on every interactive element.
  • Main shortcuts (N, Esc, Enter) work.
  • Light theme tested — not a mere color inversion.
  • AA contrast on text, AAA on critical data.
  • Dense tables have row height ≥ 44px.
  • Progress bar animates on change (300ms ease-out).
  • Empty state has a clear CTA.
  • Error messages are specific, not generic.
  • Import flow: no operation is persisted before Step 3 confirmation.
  • PDF: note number is checked before processing — an already-imported note halts the flow at Step 1.
  • Already-imported state shows number, import date, and operation count.
  • Residual import conflicts require explicit user decision — never auto-resolved.
  • Step 2 of import shows note identification on the table header.
  • Step 2 of import shows projected fiscal impact in the footer.
  • Operations imported via PDF show the note number in a hover tooltip.
  • The note-number column on the Lançamentos table is hidden by default and toggleable.
  • CSV export includes the note-number field for round-trip traceability.
  • File-read errors describe the specific problem, not a generic message.

12 · Out of scope (v1)

To avoid scope creep, the following are explicitly excluded from v1:

  • Responsiveness below 960px: handled in v1.1.
  • User theme customization: only the light/dark toggle.
  • Advanced charts (candles, heatmaps, IBOV comparison): v2.
  • Three-dimensional / decorative animations: explicitly rejected — they conflict with the fiscal-clarity principle.
  • Authentication and multi-user: v2.
